## 1. Goals & guardrails

**Goal:** every screen feels like one coherent app — Instagram-flavored per BACKBONE §9 — and works well as a genuine desktop layout, not just a stretched mobile view, without regressing any iOS/PWA behavior already built.

**Non-goals (hard guardrails, per CLAUDE.md):**

## 2. Current-state audit

Read every screen (`AuthScreen`, `ChatList`, `ChatView`, `FriendsScreen`, `NewGroupScreen`, `Profile`, `GalleryScreen`, `ChatGallery`, `MediaViewer`, `MediaBubble`, `App.tsx`) plus `index.css`. Findings:

- **No desktop layout exists.** Every screen is `min-h-[100dvh] flex-col` with no max-width wrapper (`AuthScreen` is the sole exception, capped at `max-w-sm`). On a wide viewport the whole app is one giant full-bleed column.
- **Zero design tokens.** `index.css` is just `@import 'tailwindcss'` with no `@theme` block. Every component independently hardcodes `indigo-600` / `neutral-*` / `black/10` utility strings. No single place to retheme from, and small inconsistencies have already crept in (e.g. spacing/radius values vary screen to screen).
- **Icons are emoji**, not a real icon set — inconsistent rendering risk across platforms (relevant per CLAUDE.md's platform-reality section).
- **Navigation is a single discriminated-union `View` in `App.tsx`** (`{ name: 'chats' } | { name: 'chat', chat, jumpToMessageId? } | ...`), one view rendered at a time, no router. This is fine for mobile but can't express "list pane + open chat pane visible simultaneously" as-is — it needs to grow, not be replaced.
- **Gallery is a fixed 3-column square grid** (`ChatGallery.tsx`), tiles force-cropped via `aspect-square` + `object-cover` — portrait/landscape media all get mangled to the same shape. `GalleryScreen.tsx` (top-level albums) has the same issue.
- **`MediaViewer` has no gestures** — prev/next are click-only arrow buttons; no swipe, pinch-zoom, or drag-to-close. Confirmed acceptable to add hand-rolled Pointer Events gestures (see UI-6).
- Every screen's header/back-button/spacing pattern is copy-pasted per-component rather than shared — a source of the "pages aren't coherent with each other" feeling.

## 3. Reference points taken from Mosaic (`C:\Things\Projects\mosaic\web`)

Investigated for design/implementation patterns only — Mosaic is a booru browser with different requirements (floating draggable windows, related-post side panels), so only the *techniques* below transfer, not the specific UI:

1. **Design tokens as CSS custom properties**, `:root`/`[data-theme]` scoped, light+dark parallel palettes, `color-mix()` for translucent surfaces. One file (`global.css`) is what makes dozens of components look coherent — most surfaces just reuse 4-5 tokens (`--bg-surface`, `--glass-border`, `--shadow-soft`, `--radius-md`).
2. **One JS breakpoint hook drives structural component swaps; CSS media queries at the same pixel value drive cosmetic-only sizing.** `useIsMobile()` (`matchMedia`, includes a `pointer:coarse` clause for landscape phones) — components branch their *component tree* on this, not just their styling.
3. **Masonry gallery**: predicts item height from width/height metadata before the image loads (no pop-in), packs each item into whichever column is currently shortest (real Pinterest-style packing, not CSS `column-count`, which breaks reading order), ~40 lines of layout code. Windowing/virtualization only kicks in past 120 items — irrelevant at Den's friend-group media volume, skip it.
4. **Full-screen viewer gestures**: ~180 lines of raw Pointer Events, no library. Swipe threshold is distance-OR-velocity (fast short flick counts even under the distance threshold); pinch-zoom tracks two pointer IDs in a `Map`; double-tap uses a manual timing+distance window; `setPointerCapture` so drags that leave the media element's bounds don't drop the gesture.
5. **lucide-react** everywhere, sized via a `size` prop, colored via `currentColor` so icon color follows button/text state rather than being set individually.

Not porting: floating/resizable/draggable windows, multi-panel related-post system, panel masonry, windowing/virtualization — all overkill for Den's scale and out of scope for a friend-circle chat app.

## 4. Target architecture

### 4.1 Design tokens

New file `app/src/index.css` gets a Tailwind v4 `@theme` block (or a parallel `tokens.css` imported before it — decide at implementation time) defining, at minimum:

- Surface colors: `--surface`, `--surface-raised`, `--surface-sunken` (light + dark)
- Text colors: `--text-primary`, `--text-secondary`, `--text-muted`
- Border: `--border` (replaces ad-hoc `black/10` / `white/10`)
- Accent: keep `indigo-600` as the brand accent (already used everywhere, no reason to churn it) but name it `--accent` / `--accent-hover` so it's changeable in one place
- Radius scale: `--radius-sm` (8px) / `--radius-md` (12px) / `--radius-lg` (16px) / `--radius-pill` (999px) — replacing today's inconsistent `rounded-lg`/`rounded-xl`/`rounded-2xl` mix
- Shadow: `--shadow-soft`, `--shadow-strong` (mostly for the floating media viewer / dropdowns)

Existing Tailwind `dark:` (`prefers-color-scheme`) stays as the mechanism — no manual theme toggle is in MVP scope, this is just formalizing the palette, not adding a light/dark switch UI.

### 4.2 Responsive shell

New hook `app/src/hooks/useIsMobile.ts`:
```ts
const MOBILE_MAX_WIDTH = 768; // keep in sync with the CSS media query below
```
(768 rather than Mosaic's 720 — Tailwind's own `md:` breakpoint is 768, so component *and* CSS-utility breakpoints line up instead of introducing a second magic number.)

`App.tsx`'s `AuthedApp` restructures around this:
- **Mobile (`isMobile === true`, current behavior preserved):** single view at a time, bottom tab bar (Chats/Gallery/Profile), exactly what exists today.
- **Desktop (`isMobile === false`):** left icon rail (replaces bottom tabs) + content area. For the Chats tab specifically, content area becomes **two panes**: conversation list (fixed width, ~360px) on the left, active chat on the right. This requires `View` to grow an optional "what's open in the right pane while list is visible" concept for desktop — simplest approach: keep `View` as-is for mobile navigation semantics, and on desktop keep `chats`-mode always rendering `ChatList` in the left pane regardless of `view.name`, with the right pane rendering `ChatView`/empty-state based on `view`. Gallery and Profile tabs stay single-pane (no natural second pane) on both layouts.
- Friends/NewGroup screens: on mobile these push over the whole screen (current behavior). On desktop these render as a centered modal/sheet over the shell rather than replacing the list pane — avoids losing the open chat when you go add a friend.

### 4.3 Navigation model change (the one real structural risk in this plan)

Today `AuthedApp` renders exactly one `View` full-screen. Moving to dual-pane means the chat list and the open chat must be able to render *simultaneously* on desktop. Plan: keep the existing `View` union as the single source of truth for "what's active," but let the render logic consult `isMobile` to decide whether that translates to "replace everything" (mobile) or "fill the right pane, keep the list mounted" (desktop). No new state shape needed — this is a render-logic change in `AuthedApp`, not a data model change. Flag this as the part most likely to surface edge cases (e.g. gallery-from-chat navigation, jump-to-message) — budget real testing time here.

**UI-1 implementation notes (decisions not already specified above):**
- **Token file structure:** no separate `tokens.css` — everything lives in `app/src/index.css`. Raw values (`--surface`, `--text-primary`, `--border`, `--accent`, etc.) are plain custom properties on `:root`, overridden inside `@media (prefers-color-scheme: dark) { :root { ... } }`. A separate `@theme` block then re-points Tailwind's own theme keys at those raw vars (e.g. `--color-surface: var(--surface)`) so the tokens are usable as `bg-surface`, `border-border`, `bg-accent`, `text-text-primary`, etc. `--accent`/`--accent-hover` alias Tailwind's own `--color-indigo-600`/`--color-indigo-700` rather than hardcoding hex, so the brand hue can never drift out of sync.
- **Radius scale is a rename, not a new value:** Tailwind v4's stock `rounded-lg`/`rounded-xl`/`rounded-2xl` already computed to 8/12/16px, which is exactly the sm/md/lg scale this stage wanted — so `@theme` overrides Tailwind's own `--radius-sm`/`--radius-md`/`--radius-lg` keys to 8/12/16px and every existing `rounded-lg→rounded-sm`, `rounded-xl→rounded-md`, `rounded-2xl→rounded-lg` was renamed 1:1 (zero pixel change). A new `--radius-pill: 999px` was added and every `rounded-full` renamed to `rounded-pill` (also zero visual change — both exceed any element's half-dimension). This was applied to every component in `app/src`, including the Stage-0 debug panels (`PushPoc`, `VoicePoc`, `WsProbe`, `InstallInstructions`), not just the 5 header screens, since leaving any `rounded-lg` un-renamed would have silently become 16px once the token override landed.
- **Color tokens are defined but not force-migrated everywhere.** The checklist item that's mandatory sitewide is radius; colors were only required to be *defined*. To keep this pass's diff to presentation-only with no risk of regressing the existing `text-indigo-600 dark:text-indigo-400` accent-text pattern (which intentionally differs from the plain `bg-indigo-600` used on solid buttons), existing indigo-600 usages were left as literal Tailwind color utilities. The new tokens are available for UI-3/UI-4 to adopt more thoroughly when those stages restyle bubbles/cards.
- **`ScreenHeader` back button:** uses a `lucide-react` `ArrowLeft` icon in place of the literal `←` character, but keeps the visible "Back" label — a pure glyph swap, not a redesign, per the "preserve every existing behavior" instruction.
- **`ChatView`'s stop/mic buttons went icon-only** (`Square`/`Mic`, no text label) since the aria-labels already carry "Stop recording"/"Record voice message" and the prior mic button was already icon-only (🎤 alone); this keeps both button states visually consistent.
- **`ChatList`'s media-preview labels** (`📷 Photo`/`🎥 Video`/`🎤 Voice message`, found via the grep sweep, not in the original explicit list) were converted from a plain string to a small inline `<span>` with a `Camera`/`Video`/`Mic` icon + text, since they were emoji usage the "replace every emoji usage" instruction covers.
- **App.tsx:** left untouched except for the one `rounded-2xl→rounded-lg` radius rename (`DebugTools` panel) — its `▲`/`▼` disclosure glyphs and navigation/view-stack logic were not touched, per the "don't touch App.tsx's navigation logic" guardrail and since they weren't in the emoji/glyph list this stage targeted.

**UI-2 implementation notes (decisions not already specified above):**
- **`useIsMobile`** (`app/src/hooks/useIsMobile.ts`) subscribes to `matchMedia('(max-width: 768px)').addEventListener('change', …)` directly rather than a `resize` listener + manual width check, per §4.2's guidance — this avoids re-running on every resize pixel and only fires when the boolean actually flips. Initial state is read synchronously via `useState(() => window.matchMedia(QUERY).matches)` so there's no first-render flash of the wrong layout, with a redundant re-sync inside the effect for correctness. Only Mosaic's plain width clause was ported — the `pointer: coarse` landscape-phone clause mentioned in §3.2 was left out since Den has no landscape-specific layout need yet; can be added later if a real landscape-tablet/phone case shows up.
- **Icon rail:** `w-16` (64px), full-height, `border-r border-border`, `bg-surface-raised`, live at the far left of the whole shell (outside both the list pane and the Friends/NewGroup overlay, which sits on top of everything including the rail — a `fixed inset-0 z-50` backdrop, not scoped to the content area only). Same three destinations/icons as the bottom tab bar: `MessageCircle` (Chats), `Images` (Gallery), `User` (Profile), from lucide-react, `size={22}`, active state uses the `--accent` token (`bg-accent text-white`) rather than the bottom-tab-bar's literal `text-indigo-600` — new desktop-only UI, so token-based per the "reuse tokens for new code" guidance, while the mobile `TabButton` was left with its original literal indigo classes untouched (pixel-identical mobile requirement).
- **Dual-pane Chats view:** list pane is a fixed `w-[360px]` column (per §4.2's "~360px"), `border-r border-border`, `ChatList` mounted inside unconditionally whenever `view.name` is `'chats' | 'chat' | 'friends' | 'newGroup'` (the whole "Chats tab family") so it stays mounted while a chat is open *and* while the Friends/NewGroup overlay sits on top of it. Right pane fills the remainder and renders `ChatView` keyed by `chat.id` (see below) when a chat is selected, or a plain "Select a chat to start messaging" empty state otherwise.
- **Overlay-preserves-open-chat, without a second nav-state shape:** `view` stays the single source of truth for "what's on top" exactly as instructed, but a small additional piece of state — `lastChatView` (the most recently opened `{name:'chat',…}` view) — is tracked purely as a *rendering cache* for "what the right pane should show behind the overlay" when `view.name` is `'friends'`/`'newGroup'`. It's set whenever a chat is opened (`openChat()`, used by `ChatList`, `FriendsScreen.onMessage`, `NewGroupScreen.onCreated`, and jump-to-message) and is *not* cleared when the user explicitly deselects (`view` → `{name:'chats'}`) — deselecting only stops the right pane from reading it, it doesn't erase the memory, so re-opening Friends right after a deselect still doesn't misbehave (there's nothing behind it either way, since the right pane's own render check is `view.name === 'chat' ? view : …`, not `lastChatView`, for the plain "chats" case). Closing the overlay (`closeOverlay()`) restores `lastChatView ?? {name:'chats'}`. This function is desktop-only — on mobile, Friends/NewGroup's `onBack` still targets `{name:'chats'}` directly, identical to the pre-UI-2 code, since mobile can only reach Friends/NewGroup from the bare chat list to begin with (`ChatList` unmounts whenever a chat is open on mobile), so there's never anything to restore.
- **`ChatView` is now keyed by `chat.id`** (`key={chat.id}` at both call sites). This wasn't needed pre-UI-2 (mobile always unmounts back to the list between chats), but dual-pane desktop introduced a new case the old code never had to handle: clicking chat B in the list while chat A is already open swaps `view` directly from one `{name:'chat'}` to another without an intervening `'chats'` state. Without the key, React would keep the same `ChatView` instance and just re-render with new props — leaking A's in-progress draft text and jump-to-message scroll state into B. The key forces a clean remount on every chat switch, on both layouts.
- **Height-fix approach:** confirmed the actual mechanism — Tailwind's `min-h-[100dvh]` combined with `flex-col` on nested ancestors doesn't cap child growth, it *adds*, because a flex item's automatic minimum size is its content's min-size unless the item's own overflow is non-visible; `ChatList`'s independent `min-h-[100dvh]` root was inflating the shared shell's `flex-1 overflow-y-auto` content slot past the viewport, pushing the sibling `<nav>` off-screen. Fix: `AuthedApp` now wraps every branch (mobile and desktop alike) in exactly one outermost `<div className="h-[100dvh]">` (a definite height, not just a minimum, to rule out any stacking ambiguity going forward), and `ChatView`, `ChatList`, `FriendsScreen`, `NewGroupScreen`, and `ChatGallery` all had their own root changed from `min-h-[100dvh] flex-col` to `h-full flex-col` — they fill whatever container they're mounted in now (the mobile single-view slot, the desktop right pane, the desktop single-pane content area, or the Friends/NewGroup overlay's `max-h-[85vh]` modal card) instead of asserting their own viewport-height minimum. Verified this resolves to a definite height at every mount point by tracing the ancestor chain (flex row with `align-items: stretch` under the definite-height outer wrapper for the desktop panes; `max-height` clamping the auto-height overlay card, which is a standard, well-supported pattern for "capped modal with an internally-scrolling body").
- **Gallery/Profile tabs inside the desktop shell:** wrapped in a plain `h-full overflow-y-auto` div (matching the scroll behavior the mobile shared-shell's content area already gave them) since neither `GalleryScreen` nor `ProfileTab` owns its own scroll container — no header or other structural change added to either, since that's UI-4/UI-5 scope, not UI-2's.
- **`ChatGallery` on desktop is single-pane, not an overlay and not dual-pane** — it replaces the whole content area next to the icon rail (list pane included), matching its pre-existing `onBack` target of `{name:'gallery'}` regardless of whether it was opened from the Gallery tab or from an open chat's "Gallery" button. This preserves an existing pre-UI-2 quirk (opening a chat's gallery and pressing back lands on the Gallery tab, not back on the chat) exactly as-is — not in scope to fix here.
- **Known limitation, not fully solved:** the §5 verify item ("resize across the breakpoint with an open chat, confirm no state loss") is flagged as a real risk rather than guaranteed safe. Crossing the breakpoint switches `AuthedApp` between two structurally different JSX trees (mobile's flat single-slot vs desktop's rail+dual-pane), so `ChatView` sits at a different position in the tree either way and React will remount it — losing in-progress draft text and scroll position on that specific transition. This is an inherent consequence of "one JS breakpoint hook drives structural component swaps" (§4.2, matching Mosaic's technique) rather than something CSS-only responsive design would hit; a fix would mean keeping `ChatView` at an identical tree position across both layouts, which isn't achievable given how differently the two shells are shaped. Flagging for the human resize test rather than papering over it.
- `npm run typecheck` and `npm run lint` both pass clean from repo root after this stage.
- Desktop-browser click-through self-test and the real Samsung/iPhone pass are still pending (per §6/§7 — no browser automation available in this environment). Jump-to-message and gallery-to-chat navigation (Stage 4 flows, called out in §7 as highest-risk) were traced by reading `App.tsx`/`ChatView.tsx`/`ChatGallery.tsx` together rather than exercised in a browser — worth deliberately re-testing those two flows first during the human pass.

- **`hover:`/`active:` split needs no manual media-query gating.** Confirmed by reading the installed `tailwindcss` v4 package: its `hover` variant already compiles to `@media (hover: hover) { &:hover {...} }` (a Tailwind 3.4+ default carried into v4), so every existing and newly-added `hover:` utility already only fires on real pointer devices and can never get "stuck" after a mobile tap. Incoming-bubble background moved from the ad-hoc `bg-black/5 dark:bg-white/10` + `text-neutral-900 dark:text-neutral-100` pair to the token equivalents `bg-surface-sunken` + `text-text-primary` (same computed colors, now themeable from one place). **UI-5 implementation notes (decisions not already specified above):**
- **Masonry algorithm** (`app/src/lib/masonry.ts`, new file): a plain `computeMasonryLayout(items, containerWidth, columnCount, gap)` function, no React involved — easy to hand-trace/unit-test in isolation. Splits `containerWidth` into `columnCount` equal-width columns (`columnWidth = (containerWidth - gap*(columnCount-1)) / columnCount`), then walks `items` in order; for each item it computes `ratio = width/height` (or the `4/3` fallback when either is `null`), derives `tileHeight = columnWidth / ratio`, and places the item at the top of whichever column currently has the smallest accumulated height (linear scan over `columnHeights`, ties break to the lowest column index — deterministic). Returns absolute `{left, top, width, height}` per tile plus a `containerHeight` (tallest column, minus one trailing gap) so the caller can size the relatively-positioned wrapper. This is real shortest-column packing, not CSS `column-count` — traced by hand with a 390px/3-column/2px-gap example mixing portrait, landscape, null-dimension, and square items to confirm columns fill left-to-right first pass and then balance correctly on subsequent items (each item lands in the column that's actually shortest at that point, not round-robin). Rendering: `ChatGallery`'s grid items are positioned with inline `left/top/width/height` styles inside one `relative` wrapper `div` sized to `containerHeight` — chosen over a flex-column-per-column approach because absolute positioning keeps the DOM structure flat (one array of tiles, easy to key/map) and makes the "which column" detail purely a positioning concern, not a component-tree concern.
- **Column count / width measurement** (`app/src/hooks/useElementWidth.ts`, new file): a `ResizeObserver`-backed hook that measures the *actual* container's rendered width, not a `useIsMobile()` boolean. Reasoning: `useIsMobile` is a binary device-class signal (≤768px vs not), but desktop's single-pane gallery content area (full width minus the 64px icon rail, no max-width wrapper anywhere in the shell) ranges continuously from ~500px on a small laptop to 1400px+ on a wide monitor — a single "desktop = N columns" constant would either starve a small desktop window or under-fill a large one. Measuring the real container width and re-deriving column count from it handles both the mobile/desktop split *and* arbitrary desktop window widths with one mechanism. The hook measures inside `useLayoutEffect` (not `useEffect`) specifically so the first real width commits before the browser paints — avoids a visible flash from the 0-width fallback layout on mount. `galleryColumnCount()` (media grid) buckets at 480/700/960px → 3/4/5/6 columns, tuned to keep tiles roughly 110-160px wide; the <480px bucket reproduces the original fixed 3-column mobile grid exactly. `albumColumnCount()` uses the same breakpoints but one column fewer at each step (2/3/4/5) since album tiles carry a text footer and read better a bit larger.
- **Null-dimension fallback:** items with `width === null || height === null` (still-processing media — rare, per §7) use a constant `4/3` aspect ratio in `computeMasonryLayout`, matching Mosaic's convention named in §3.3/§7. No special-casing needed beyond the ternary in the ratio calculation — the item still participates in normal shortest-column packing, it just gets a generic landscape-ish box instead of a predicted one.

**UI-6 implementation notes (decisions not already specified above):**
- **Gesture surface is the `<img>` element only, not the whole viewer or `<video>`.** Two separate scoping decisions, both deliberate: (1) the pointer handlers live on the image itself rather than the full-screen backdrop, so the pre-existing click-to-close-on-letterbox-tap behavior (tapping the black area around the image, handled by the outer `fixed inset-0` div's `onClick={onClose}`) stays completely untouched — it's a plain `click`, unrelated to the new pointer-gesture code path. (2) `<video>` was deliberately **not** given the gesture layer at all — it keeps its exact pre-UI-6 rendering (native `controls`, `autoPlay`, `onClick` stopPropagation, nothing else). Reasoning: native video `controls` render their own touch surface (scrubber, play/pause, fullscreen button) inside the element's box; attaching `setPointerCapture`-based custom gestures to the same element risks hijacking that native surface (e.g. a tap intended for the scrubber getting captured as a would-be double-tap or swipe), and that specific interaction (does the scrubber still drag correctly under a competing pointer-capture layer) cannot be verified without a real touchscreen. Given "preserve every existing behavior exactly" as a hard constraint and no way to confirm video controls still work if gestures were added, the safer, explicitly-scoped choice was to ship swipe/pinch/double-tap for images only and leave video on arrow-button + tap-outside-to-close navigation, matching its behavior before this stage. Logged as a real, known limitation in §8 below, not silently decided. **Superseded 2026-07-20:** video now gets swipe-nav/swipe-close (not pinch/double-tap), gated by a bottom exclusion zone rather than left off entirely — see the §8 follow-up entry for the implementation.
- **State shape:** one `useState<{scale,x,y}>` for the rendered transform (`transform: translate(x,y) scale(s)` — this order matters: it makes `x`/`y` plain screen-pixel deltas independent of zoom level, so a drag tracks 1:1 under the finger whether zoomed or not) plus a boolean `interacting` (disables the CSS transition while a gesture is live, re-enables it for a smooth snap-back once released). Everything else — the live pointer map, the in-progress gesture's bookkeeping, and the last-tap timestamp/position for double-tap detection — lives in refs (`pointersRef: Map<pointerId, {x,y}>`, `gestureRef: GestureState | null`, `lastTapRef`), not state, since none of it needs to trigger a re-render by itself (only the derived `transform`/`interacting` writes do). The four pointer handlers (`onPointerDown/Move/Up/Cancel`) are plain functions redefined every render (not wrapped in `useCallback`) specifically so they always close over the current `transform` state and current `onPrev`/`onNext`/`onClose` props — the task flagged "stale closures over pointer maps" as a bug class to watch for (per the `useElementWidth` cautionary tale), and the fix here is structural: refs can never be stale by definition, and the handler closures are cheap to recreate each render so there was no reason to introduce a memoization layer that could reintroduce staleness.
- **Distinguishing swipe vs. pinch vs. double-tap vs. an ordinary click on a control:** driven entirely by `pointersRef.current.size` at `pointerdown` time (1 pointer → drag/tap/double-tap candidate; 2 → pinch; 3+ → ignored, the existing 2-pointer gesture keeps tracking) plus one `axis: 'horizontal'|'vertical'|null` lock that's only set once movement clears `MOVE_TOLERANCE` (10px) — so a genuine tap (never crosses that threshold) leaves `axis` null and resolves to a no-op at `pointerup`, never a swipe. Double-tap is detected independently of the drag/axis machinery: at `pointerdown`, if a previous tap was recorded within 300ms and within 30px (both checked against `lastTapRef`), the gesture is flagged `isDoubleTap`; at `pointerup`, that flag only fires the zoom toggle if the pointer *also* didn't move past `MOVE_TOLERANCE` in between (`movedEnough`, computed via `Math.hypot(dx,dy)` so it's correct in both the not-zoomed axis-lock branch and the zoomed-pan branch, which never touches `axis` at all — an early draft of this reused `axis !== null` as the "did it move" shortcut and that's wrong exactly in the zoomed case, since panning never sets `axis`; switched to a direct distance check to avoid that bug). A real click on a chrome button (close/jump/prev/next) never reaches any of this — those are separate DOM elements outside the `<img>`, so their own `onClick` handlers fire completely independently; `setPointerCapture` on the image means a drag that happens to pass *over* a button's screen position mid-gesture does not spuriously click it (capture re-targets all events for that pointer id to the image regardless of what's visually underneath), which is the intended, standard reason to use pointer capture here.
- **Swipe threshold — distance OR velocity:** `SWIPE_DISTANCE_THRESHOLD = 60px` or `SWIPE_VELOCITY_THRESHOLD = 0.5px/ms` (500px/s) for prev/next; `CLOSE_DISTANCE_THRESHOLD = 100px` or `CLOSE_VELOCITY_THRESHOLD = 0.5px/ms` for swipe-down-to-close (close needs a bit more distance than a horizontal swipe so an incidental vertical wobble mid-horizontal-drag can't also register as a close — the two are still mutually exclusive regardless, since `axis` locks to whichever direction dominates at the moment the drag first clears `MOVE_TOLERANCE`). Velocity is `abs(delta)/elapsed-ms` measured from the whole gesture's start to end (not a rolling average) — simple and sufficient for "did they flick it," matching the plan's explicit ask that a fast short flick count even under the distance threshold. Values were picked from typical mobile-swipe conventions (not measured against a real device, since none is available in this environment) and are called out in §8 as worth tuning after the real-device pass if they feel off.
- **Pinch:** tracks two pointer IDs in `pointersRef` (a plain `Map`, per the plan's suggested shape); scale is `baseScale * (currentDistance / startDistance)` recomputed from the live map on every `pointermove` for either finger, clamped to `[1, 4]`. The pinch midpoint is tracked too, and its drift is added to the translate — a two-finger pinch that also drags sideways pans the image at the same time, which is the conventional combined pinch-zoom+pan gesture and was cheap to add once the midpoint was already being computed for the distance calc. Lifting one finger mid-pinch re-anchors to the remaining finger's last known position and downgrades to single-pointer drag/pan with no visual jump (verified by hand-tracing: the new drag gesture's `baseX/baseY` are seeded from the *current* `transform`, not reset to 0).
- **Double-tap:** manual timing+distance window (300ms, 30px) compared against the previous `pointerdown`, not the previous `pointerup` — matches how real double-tap detection works (two down-events close together), and naturally handles "tap, drag a little, release, tap again in the same spot" as *not* a double-tap (the second `pointerdown`'s position is what's compared, and a completed drag doesn't move `lastTapRef`'s recorded position since that's set at `pointerdown`, not updated during drag). Zoom toggles between `scale=1` and a fixed `DOUBLE_TAP_SCALE=2.5`, always centered (`x=0,y=0`) rather than anchored to the tap point — deliberate simplification: correctly anchoring a `translate()·scale()` zoom to an arbitrary tap coordinate needs transform-order math that's easy to get subtly wrong (translate is applied in *post-scale* pixel space in `translate() scale()` order, which is what makes plain-pixel-delta panning correct elsewhere in this same gesture code, but complicates anchor-point math for the initial jump-to-zoom step) and there's no way to visually confirm the anchor math is right without a real device. A centered zoom is standard, unambiguous UX and avoids that whole bug class.
- **Zoom/pan reset on navigation:** one `useEffect` keyed on `media.id` resets `transform` to `{scale:1,x:0,y:0}`, clears `pointersRef`/`gestureRef`/`lastTapRef`, and resets `interacting` to `false`. This is necessary because `MediaViewer` is *not* remounted on prev/next (`ChatGallery`/`ChatView` render it conditionally on a truthy media value, no `key` prop) — only the `media` prop changes — so without this effect a zoomed-in state would carry over to the next item exactly the way UI-2's `ChatView` remount bug worked in reverse (there, a missing `key` leaked state across chats; here, the fix is the opposite instinct — an effect keyed on the identity that *should* reset state, since adding a `key={media.id}` to `MediaViewer` itself would work too but would also discard the backdrop's own mount-transition and drop pointer capture mid-gesture on programmatic navigation, which the effect approach avoids). The effect also runs once on initial mount (harmless — values already match the initial `useState`).
- **Pan clamping:** while zoomed, pan (both single-finger drag and pinch-drift) is loosely clamped so the image can't be dragged/pinched completely off-screen with no way back (`clampTranslate`, bounding translate to half the zoomed overflow relative to the image's own unzoomed box — a heuristic, not pixel-exact "keep 1px visible" math, but enough that double-tap-to-reset is always reachable). Not an explicit checklist item, but a direct, cheap mitigation for a real "lost the image" dead-end that hand-tracing the pinch/pan math surfaced.
- **iOS edge-swipe-back / scroll interference (§7's carried-forward risk):** the `<img>` gesture surface gets `touchAction: 'none'` (not just `manipulation`) plus `preventDefault()` in the pointermove handler (guarded by `e.cancelable`), so a touch that starts on the image is claimed entirely by the custom gesture code and the browser never gets a chance to interpret the same touch as a scroll or an edge-swipe-back. `draggable={false}` + `onDragStart` preventDefault on the `<img>` also blocks the browser's native HTML5 image-drag (a real, separate gotcha from Pointer Events — without it, a desktop mouse-drag on an `<img>` can kick off native drag-ghost behavior that interferes with `pointermove` tracking). This mitigation is scoped to the image element specifically (see the surface-scoping note above) — the letterbox area and the video element don't get the same hardening, which is an accepted gap given those areas don't need custom gesture handling in the first place (letterbox: plain click-to-close, already had no scroll/swipe behavior to protect; video: native controls already handle their own touch interactions correctly without any change from this stage).

**UI-7 decisions (asked and answered before implementation):**
- **Waveform data is computed client-side, not stored.** Three options were put to the user — client-side WebAudio decode, a deterministic fake pattern, or a server-computed `peaks` column via ffmpeg. The user chose client-side decode, with the added instruction that the bars be *mirrored and centered* (Instagram-style) rather than baseline-anchored. This keeps the stage presentation-only: no migration, no `MediaInfo` change, nothing for old voice messages to backfill.
- **Peaks load on first play, not on mount.** Decoding on mount would download every voice note in an open chat. Instead each bubble draws a deterministic ghosted placeholder pattern (seeded from the media id via FNV-1a + xorshift32, so it's stable across reloads and never shuffles), and swaps in real peaks after the first play. Real peaks render at full opacity, the placeholder is ghosted throughout — it never claims to be data it isn't. Results are cached module-level by media id so scrolling away and back doesn't re-decode, with an in-flight map so a double-tap can't start two fetches.
- **Decoding uses `OfflineAudioContext`, and `play()` is never awaited behind it.** ⚠️ iOS: a live `AudioContext` needs a user gesture and a `resume()`; an OfflineAudioContext used purely as a decoder needs neither. `togglePlay` calls `audio.play()` synchronously first and *then* fires the decode — awaiting the fetch before `play()` would break the gesture association and Safari would refuse to start playback. Both the promise and the legacy callback form of `decodeAudioData` are honored (iOS 16.4 is our floor and has the promise form; the fallback is three lines and removes a whole class of "works on Android, silently no bars on iOS" bug). A failed decode resolves `null` and simply leaves the placeholder up — playback is a separate `<audio>` element and never depends on any of it.
- **Everything in the player draws in `currentColor`** at varying opacity, so one component sits correctly in both the accent-filled "mine" bubble and the sunken "theirs" bubble with no ownership branch. Mirroring is free: bars are vertically centered by the flex row, so growing a bar's height extends it equally up and down — there is no second mirrored copy to keep in sync.
- **The tail is a corner, and only the run's last bubble has one.** Every bubble is uniformly `rounded-lg`; the tail is `rounded-br-[4px]` / `rounded-bl-[4px]` on the sender's side, applied only when the block is last in its run. Bubbles inside a run sit 2px apart (runs are 12px apart), so a burst reads as one connected block. Runs break on sender change or a >5min gap (`RUN_WINDOW_MS`); a group chat's sender label is now printed once per run instead of inside every bubble.
- **Media gets no bubble.** `bare = media !== null && kind !== 'voice'` — photos, videos, and their processing/failed placeholders (already self-contained cards) render with nothing behind them. A caption becomes its own small bubble underneath rather than padded text inside a container wrapping the image. Voice is the deliberate exception and stays in a bubble.
- **A stack is a drawing, never an addressable unit.** Each card is still its own message on the wire. Only *bare* ready photos/videos stack — a caption makes an item a standalone thought, and processing/failed/`pending:` items have no thumbnail and (for pending) no server-side id to act on. Two safeguards keep this from colliding with Stage-6 deletion: entering multi-select turns stacking **off** entirely (`groupMessages(messages, { stack: !selectionMode })`), so every message is individually selectable; and long-pressing a stack skips the single-message action sheet (Copy/Delete are per-message) in favour of selecting all of its messages, which also expands the stack into the individual bubbles the user just selected. There is no code path that can produce an action scoped to "the stack".
- **Multi-pick uploads run sequentially, and the caption rides the first item only.** The media pipeline is per-item regardless, a serial queue keeps the progress bar honest ("Uploading photo 2 of 3… 40%"), and it avoids N concurrent PUTs on a phone radio. Partial failure is reported as such ("2 of 3 uploads failed") rather than collapsing to a single generic error. Repeating the composer text on every message of a batch would read as spam, hence first-only.
- **Tagging stays a viewing-time action** (the user's explicit choice over adding tag entry to the composer) — the send path is untouched. What changed is reach: the existing `TagEditor` is now mounted in the chat-side viewer, so tags no longer require a detour through the gallery screen. The viewer also gained prev/next when opened from a stack's grid sheet, reusing the same `MediaViewer` the gallery uses.
- **One new endpoint: `GET /media/:id/tags`** (`MediaTagsResponse` in `shared/src/api.ts`). The gallery batches tags into its page response; the chat-side viewer opens straight off a bubble with no page behind it. Same `assertMember` gate, same data already exposed via the gallery — no widening of visibility, no schema change. Client side it's a `['mediaTags', id]` query, added alongside `['gallery']` in the WS `tag.added`/`tag.removed` invalidation so shared-wiki edits stay live in both places.
- **⚠️ Nothing in this stage was seen rendered.** `npm run typecheck`, `npm run lint`, and `npm run app build` all pass, and the generated CSS was grepped to confirm the new utilities emit correctly (`bg-current/15` compiles to a `color-mix` with a solid `currentColor` fallback; `rounded-br-[4px]`/`rounded-bl-[4px]` emit). But a throwaway preview harness could not be rendered: the Vite dev server's esbuild died with `runtime: cannot allocate memory` and a subsequent build crashed with a stack-buffer-overrun exit — machine memory exhaustion, not a code fault. So the *layout* claims here (fanned-card geometry, waveform bar sizing, tail contrast, bare-media alignment) are reasoned, not observed. This stage needs a desktop click-through before it's trusted, on top of the standing device pass.

### Cleanup pass notes: fixing the UI-2 resize/remount bug (2026-07-20)

**Direction chosen: A (state lifting), not B (unify the render tree).** Direction B (keep `ChatView` at one stable JSX position regardless of `isMobile`, expressing mobile/desktop differences as CSS around it) was read in full and rejected for this codebase, not attempted and abandoned partway. Reasoning: mobile and desktop aren't just differently-styled versions of the same layout — they're different *navigation paradigms*. Mobile is a single-slot stack (`AuthedApp`'s mobile branch renders exactly one of `FriendsScreen` / `NewGroupScreen` / `ChatView` / `ChatGallery` / the tabbed `{ChatList | GalleryScreen | ProfileTab} + BottomNav` block, full-screen, with everything else unmounted) while desktop is a persistent icon rail + dual-pane shell with a modal overlay on top. To get `ChatView` to sit at an identical ancestor chain in both, essentially every mobile-only full-screen state (`friends`, `newGroup`, `chatGallery`, the tab-bar block) would have needed to become an always-mounted, CSS-hidden sibling inside the desktop shell's shape instead of a conditional branch — a rewrite of most of `AuthedApp`, not a targeted one. That's exactly the fragility the task warned about ("accidentally wrapping `ChatView` in a conditional that itself changes element type between branches would silently reintroduce the bug"), on the single highest-traffic file in the app, with jump-to-message, gallery-to-chat navigation, and the `lastChatView` overlay restore all riding on the same render logic. Direction A is a much smaller, more locally-reasoned diff, so it's what shipped.

**What's fixed:** draft text. `AuthedApp` (`app/src/App.tsx`) now owns a `draftCacheRef = useRef(new Map<string, string>())` — a plain per-chat `Map`, not `useState`, since writes happen on every keystroke and must not trigger an `AuthedApp` re-render (`ChatView` already re-renders itself locally via its own state; the cache only needs to be *readable* at the instant a fresh `ChatView` instance mounts). `ChatView` (`app/src/components/ChatView.tsx`) gained two props, `initialDraft: string` and `onDraftChange: (draft: string) => void`. Its `draft` state changed from `useState('')` to `useState(initialDraft)`, and the local `setDraft` was renamed to `setDraftState` with a new `setDraft(value)` wrapper that calls both `setDraftState(value)` and `onDraftChange(value)` — every existing call site (`onChange` on the composer input, `submit()`'s reset-to-empty, `runUpload()`'s reset-to-empty-on-success) still just calls `setDraft(...)` unchanged, so the diff inside `ChatView` is two lines plus the wrapper, not a rewrite of the draft-handling logic. Both `ChatView` call sites in `App.tsx` (mobile branch and desktop right pane) pass `initialDraft={draftCacheRef.current.get(chatId) ?? ''}` and `onDraftChange={(draft) => draftCacheRef.current.set(chatId, draft)}`, using the same `chat.id` each site already uses for its `key={chat.id}`.

**Verified by hand-tracing the state flow, not by running a browser:**
1. *Resize case (the reported bug):* desktop, chat A open, user types "hello" — `setDraft("hello")` fires on every keystroke, writing `draftCacheRef.current.set('A', "hello")` synchronously (not deferred to unmount). Window crosses 768px → `useIsMobile()` flips → `AuthedApp` re-renders into the mobile branch, a structurally different tree → React unmounts the desktop `ChatView` instance and mounts a new one (same `key={chat.id}`, different tree position, so no reconciliation match — this part of the original bug is unchanged and expected). The new instance's `initialDraft` prop reads `draftCacheRef.current.get('A')`, which already holds `"hello"` from before the resize (no race: the cache write is synchronous with the keystroke, not tied to the old instance's unmount). `useState(initialDraft)` seeds the new instance's `draft` state with `"hello"`. Draft survives.
2. *Genuine chat-to-chat switch (must keep working exactly as before):* chat A open with a drafted "hello", user clicks chat B in `ChatList` — `view` goes directly from `{name:'chat', chat:A}` to `{name:'chat', chat:B}`, so `key` changes from `A.id` to `B.id` and `ChatView` fully unmounts/remounts exactly as it did pre-fix (this behavior, called out explicitly as "fine and currently expected," was not touched). The new instance seeds `initialDraft` from `draftCacheRef.current.get('B')`, which is empty unless chat B was separately drafted before — no leakage of A's draft into B's composer. One incidental, positive side effect worth being explicit about: since the cache is keyed by `chat.id` and outlives any single `ChatView` instance, navigating away from A and back to A later now also restores the "hello" draft, whereas pre-fix it would have been silently lost on every chat switch, not just the resize case. This is a natural consequence of moving the source of truth for "does chat A have a pending draft" out of the component instance and into a longer-lived cache — it doesn't touch message list state, scroll position, upload state, or the `key`-driven remount itself, so it doesn't conflict with the "preserve unmount-remount-on-switch" instruction, but it is a small, deliberate behavior change beyond the literal resize-bug fix, flagged here rather than left implicit.
3. *Jump-to-message / gallery-to-chat navigation:* traced both `App.tsx`'s `jumpToMessage()` and `ChatGallery`'s `onJumpToMessage` callback. When the target chat is already the one open (`key` unchanged), `ChatView` does not remount — only the `jumpToMessageId` prop changes, which the existing scroll-to-message effect already handles; `initialDraft` is a `useState` initializer, so React ignores its new value on a non-mount render, exactly like the existing `chat`/`me` props are already re-passed every render without resetting other local state. When the target is a different chat (`key` changes), it's the same code path as point 2 above — new instance, cache-seeded draft, no special interaction with the jump/scroll logic since that logic lives entirely in `ChatView`'s own effects and doesn't read `initialDraft`/`onDraftChange` at all. Neither `App.tsx`'s navigation functions (`openChat`, `jumpToMessage`, `openGalleryFor`) nor `ChatGallery.tsx`/`ChatList.tsx` were modified by this change.
4. *Friends/NewGroup desktop overlay (`lastChatView`):* unaffected by construction — `rightPaneChat` (which can resolve to `lastChatView` while the overlay is open) still drives both the `key` and the new `initialDraft`/`onDraftChange` props from the same `chat.id`, so the chat sitting behind the overlay keeps its draft cached exactly like any other open chat; `lastChatView`'s own logic in `App.tsx` (`openChat`, `closeOverlay`) was not touched.

**What's still open (deliberately not attempted, per the task's "be honest about partial fixes" instruction):**
- **Scroll position** is not preserved across a resize-triggered remount. Restoring it correctly would mean caching `scrollTop` per chat (parallel to the draft cache) and, on mount, choosing between "scroll to bottom" (today's default whenever a chat opens) and "restore the cached position" — but that choice can't be conditioned on *why* the component is mounting (resize vs. a fresh chat-open vs. returning to a previously-viewed chat), because from `ChatView`'s perspective a mount is a mount, the same ambiguity the draft fix works around by making "the fix" simply not care why it remounted. Applying that same trick to scroll position would change the well-established "opening a chat scrolls to the bottom" behavior for *every* remount, including plain chat-switching and revisiting a chat later, not just the resize case — and it would need to interact correctly with the existing jump-to-message scroll effect (which also drives `bottomRef`/`messageRefs` scrolling and is explicitly called out as the highest-risk existing flow to protect). Given the primary, explicitly-confirmed bug (draft loss) is fully fixed and low-risk, and scroll position was scoped as "ideally" rather than required, this was left open rather than rushed.
- **Upload-in-progress** is not preserved. The underlying upload request itself is not actually cancelled by the remount — `runUpload()` in `ChatView` calls `uploadMedia()` (`app/src/lib/media.ts`), a plain `async` function with no `AbortController`/cleanup wiring to the component's lifecycle, so the in-flight `XMLHttpRequest` and the promise chain around it keep running after `ChatView` unmounts, and per `uploadMedia`'s own doc comment the resulting message "arrives over WS to every member, including this tab," so the upload itself should still complete and the message should still show up once processed. What's lost is purely the *UI feedback* — the progress bar and any upload-error state are local `useState` (`upload`, `uploadError`) with no equivalent lifted cache, so a fresh `ChatView` instance after a resize has no way to know an upload is still in flight for that chat and won't show a progress bar until the new message arrives over the socket. This was flagged by the task itself as "the hardest — the actual upload request lifecycle currently lives entirely inside `ChatView`'s closures" and was not attempted; doing it properly would mean moving the upload lifecycle (not just its progress number) out of `ChatView` into something like `AuthedApp` or a dedicated store, which is a larger architectural change than this cleanup pass's risk budget supports. Not claimed as solved.

**Not touched:** `app/src/components/ChatList.tsx`, `app/src/components/ChatGallery.tsx`, `app/src/components/FriendsScreen.tsx`, `app/src/components/NewGroupScreen.tsx`, `app/src/components/MediaViewer.tsx`, and all of `/server`/`/shared` — this was scoped to `App.tsx` + `ChatView.tsx` only, per the task's client-only, minimal-surface-area instruction. `npm run typecheck` and `npm run lint` both pass clean from repo root after this change.
